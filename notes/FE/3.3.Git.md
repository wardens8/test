# Git

## Git 简介

版本控制（Revision control）是一种在开发的过程中用于管理我们对文件、目录或工程等内容的修改历史，方便查看更改历史记录，备份以便恢复以前的版本的软件工程技术。Git 就是这样的一个工具。

- git 官网：https://git-scm.com/

## 安装及配置

- 安装：官网下载之后，全部下一步即可，其环境变量会自动配置。安装完成后，打开命令行，输入 `git -v`，能看到正常输出即表示安装成功
- 配置
  - git 设置中文方法：进入界面-右键 `options --> Windows --> UI language` 改为 `zh-CN`
  - 使用 git 前，我们需要配置一下两个属性 name 和 email，这两个信息会用来在存储代码时记录用户的身份。可以直接在命令行中通过指令来设置：

```bash
git config --global user.name "xxx"
git config --global user.email "xxx"
```

- 查看配置

```bash
git config -l # l是list列表的意思
```

- 查看不同级别的配置文件

```bash
git config --system --list # 查看系统config
git config --global --list # 查看当前用户（global）配置
```

- git 相关的配置文件位置（这里可以直接编辑配置文件，通过命令设置后会响应到这里）

```bash

C:\Users\[用户名]\.gitconfig # 只适用于当前登录用户的配置 --global 全局
```

## 文件状态

git 本地有三个工作区域：**工作目录**（Working Directory）、**暂存区**（Stage/Index）、**本地资源库**（Repository 或 Git Directory）。如果在加上**远程的 git 仓库**（Remote Directory）就可以分为四个工作区域。文件在这四个区域之间的转换关系如下：

- `Workspace` 工作区：就是你平时存放项目代码的地方。
- `Index/Stage` 暂存区：用于临时存放你的改动，事实上它只是一个文件，保存即将提交到文件列表信息。
- `Repository` 仓库区（或本地仓库）：就是安全存放数据的位置，这里面有你提交到所有版本的数据。其中 HEAD 指向最新放入仓库的版本。
- `Remote` 远程仓库：托管代码的服务器，可以简单的认为是你项目组中的一台电脑用于远程数据交换。（github、gitee、gitlab）

![](https://mmbiz.qpic.cn/mmbiz_png/uJDAUKrGC7Ksu8UlITwMlbX3kMGtZ9p0NJ4L9OPI9ia1MmibpvDd6cSddBdvrlbdEtyEOrh4CKnWVibyfCHa3lzXw/640?wx_fmt=png&tp=webp&wxfrom=5&wx_lazy=1&wx_co=1)

git 中的文件有两种状态：未跟踪和已跟踪。未跟踪指文件没有被 git 所管理，已跟踪指文件已被 git 管理。已跟踪的文件又有三种状态：暂存、未修改和已修改。

- 未跟踪 `Untracked(U)`：此文件在文件夹中，但并没有加入到 git 库，不参与版本控制。通过 `git add` 状态变为 `Staged`
- 已跟踪：git 管理的文件有三种状态：已暂存 `Staged`，未修改 `Unmodified`，已修改 `Modified`
  - 暂存 `Staged(A)`：表示文件修改已经保存，但是尚未提交到 git 仓库，处于暂存状态
  - 未修改 `Unmodified`：表示磁盘中的文件和 git 仓库中文件相同，没有修改
  - 已修改 `Modified(M)`：表示磁盘中文件已被修改，和 git 仓库中文件不同
- 可以通过 `git status` 来查看文件的状态

## 基本操作

- 默认情况下，磁盘中的文件并不由 git 管理，我们必须要对代码目录进行初始化，初始化后 git 才能正常的管理文件。
- 进入目录后，直接在目录中执行 `git init` 即可完成项目的初始化，初始化后目录中会多出一个 `.git` 目录，这个目录用来存储代码的版本信息，有了 `.git` 就意味着项目现在已经开始被git管理了，不希望项目被git管理时，只需删除项目中的 `.git` 即可

初始化仓库

- `git` 查看git全部命令
- `git init` 初始化git仓库
- `git status` 查看文件状态
- `git status -s` 查看文件状态（简写），s是short简短的意思
- `git status [filename]` 查看指定文件状态
- `git clone [repo_url]` 克隆git仓库

提交文件

- `git add [filename]` 将某文件切换至暂存状态（未跟踪/已修改—>暂存）
- `git add .` 将全部文件切换至暂存状态（未跟踪/已修改—>暂存）
- `git commit -m [file_message]` 将暂存的文件存储到仓库里并提交日志，m是message消息、日志的意思（暂存—>未修改）
- `git commit -a -m [file_message]` 提交所有已修改的文件，未跟踪的文件不会提交。就是将两步提交合在一起（已修改—>未修改）

比较文件

- `git diff` 比较工作区和暂存区的文件前后差距
- `git diff [filename]` 比较工作区和暂存区的文件前后差距
- `git diff --cached` 比较暂存区和本地库文件前后差距
- `git diff --cached [filename]` 比较暂存区和本地库文件前后差距
- `git diff HEAD` 比较工作区和本地库文件前后差距
- `git diff HEAD -- [filename]` 比较工作区和本地库文件前后差距

重置文件

- `git reset HEAD [filename]` 将暂存区的文件恢复到工作区，HEAD表示当前版本
- `git reset --soft HEAD^` 回退到上一个版本，保留工作区修改
- `git reset --hard HEAD^` 回退到上一个版本，丢弃工作区修改，HEAD^是上一个版本，HEAD^^是上两个版本，HEAD~n是上n个版本
- `git reset --hard [commit-id]` 回退到指定版本（通过 git log/reflog 查 commit id）

撤销文件

- `git restore [filename]` 恢复文件
- `git restore --staged [filename]` 取消暂存状态

删除文件

- `git rm [filename]` 删除文件
- `git rm -f [filename]` 强制删除文件，f是force强制的意思

查看历史

- `git log` 查看git的每一次操作
- `git log --pretty=oneline` 以一行的形式查看提交历史，且哈希值完整
- `git log --oneline` 以一行的形式查看提交历史，且哈希值不完整
- `git reflog` 查看所有的提交历史（包括已经删除的）
- `git reflog --pretty=oneline` 以一行的形式查看提交历史，且哈希值完整
- `git reflog --oneline` 以一行的形式查看提交历史，且哈希值不完整
- `git log --graph` 以图形的形式查看提交历史

版本回退

- `git reset --soft HEAD^` 回退到上一个版本，保留工作区修改
- `git reset --hard HEAD^` 回退到上一个版本，丢弃工作区修改，HEAD^是上一个版本，HEAD^^是上两个版本，HEAD~n是上n个版本
- `git reset --hard [commit-id]` 回退到指定版本（通过 git log/reflog 查 commit id）

操作分支

- git 在存储文件时，每一次代码代码的提交都会创建一个与之对应的节点，git 就是通过一个一个的节点来记录代码的状态的。节点会构成一个树状结构，树状结构就意味着这个树会存在分支，默认情况下仓库只有一个分支，命名为 master。在使用 git 时，可以创建多个分支，分支与分支之间相互独立，在一个分支上修改代码不会影响其他的分支
- 若合并分支时出现冲突，根据实际情况进行解决，完成之后重新进行提交即可
- `git branch` 查看所有本地分支
- `git branch -r` 查看所有远程分支，r是remote远程的意思
- `git branch <branch name>` 创建新的分支
- `git branch -M <branch name>` 修改当前分支的名字
- `git branch -d <branch name>` 删除已合并的分支，d是delete删除的意思
- `git branch -D <branch name>` 强制删除未合并的分支，d是delete删除的意思
- `git switch <branch name>` 切换分支（git 2.23.0 版本及以上）
- `git checkout <branch name>` 切换分支（git 2.23.0 版本以下）
- `git switch -c <branch name>` 创建新的分支并切换至该分支（git 2.23.0 版本及以上）
- `git checkout -b <branch name>` 创建新的分支并切换至该分支（git 2.23.0 版本以下）
- `git merge <branch name>` 合并分支
  - 冲突文件会标记
    - `<<<<<<< HEAD（当前分支内容）`
    - `=======（待合并分支内容）`
    - `>>>>>>> [branch name]`
  - 手动编辑保留需要的内容，删除冲突标记
  - 执行 git add <冲突文件> → git commit（merge 冲突无需写 -m，会自动生成提交信息）

变基

- 在开发中除了通过 merge 来合并分支外，还可以通过变基来完成分支的合并。
- 我们通过 merge 合并分支时，在提交记录中会将所有的分支创建和分支合并的过程全部都显示出来，这样当项目比较复杂，开发过程比较波折时，我必须要反复的创建、合并、删除分支。这样一来将会使得我们代码的提交记录变得极为混乱。
- 原理（变基时发生了什么）：
  1. 当我们发起变基时，git 会首先找到两条分支的最近的共同祖先
  2. 对比当前分支相对于祖先的历史提交，并且将它们提取出来存储到一个临时文件中
  3. 将当前部分指向目标的基底
  4. 以当前基底开始，重新执行历史操作
- 变基和 merge 对于合并分支来说最终的结果是一样的！但是变基会使得代码的提交记录更整洁更清晰！注意！大部分情况下合并和变基是可以互换的，但是如果分支已经提交给了远程仓库，那么这时尽量不要变基。因为变基会改写提交历史，若多人协作同一远程分支，变基后推送需强制（git push -f），易导致他人提交丢失，风险极高。
- `git rebase <branch>` 将当前分支的基底变为指定分支，出现冲突时和merge相同的解决方式

标签（tag）

- 当头指针没有指向某个分支的头部时，这种状态我们称为分离头指针（HEAD detached），分离头指针的状态下也可以操作操作代码，但是这些操作不会出现在任何的分支上，所以注意不要再分离头指针的状态下来操作仓库
  - `git switch <版本号> --detach`
- 如果非得要回到后边的节点对代码进行操作，则可以选择创建分支后再操作
  - `git switch -c <分支名> <提交id>`
- 可以为提交记录设置标签，设置标签以后，可以通过标签快速的识别出不同的开发节点：
  - `git tag` 查看全部标签
  - `git tag [标签名]` 给当前版本打标签
  - `git tag [标签名] [版本号]` 给指定版本打标签
  - `git tag -a [标签名] -m "标签说明" [commit-id]` 创建带注释的标签
  - `git show [标签名]` 查看标签详情
  - `git tag -d [标签名]` 删除指定标签
  - `git push [远程仓库] [标签名]` 将指定标签推送至远程仓库
  - `git push [远程仓库] --tags` 将全部标签推送至远程仓库
  - `git push [远程仓库] --delete [标签名]` 删除远程标签
  - `git switch [标签名]` 回到标签位置

## `.gitignore` 忽略文件

默认情况下，git 会监视项目中所有内容，但是有些内容比如 `node_modules` 目录中的内容，我们不希望它被 git 所管理。我们可以在项目目录中添加一个 `.gitignore` 文件，来设置那些需要 git 忽略的文件

在主目录下建立 `.gitignore` 文件，此文件有如下规则：

```bash
# ***        # 为注释
node_modules # 忽略 node_modules 目录
*.log        # 忽略所有 .log 结尾的文件，这样的话上传就不会被选中！
/build/      # 忽略 build/ 目录下的所有文件
doc/*.log    # 忽略 doc/ 目录下所有 .log 结尾的文件
!lib.txt     # 否定匹配，lib.txt 文件不会被忽略
```

```bash
node_modules
dist
```

> `.gitignore` 只对未跟踪文件生效：若文件已被 `git add` 跟踪，修改 `.gitignore` 后需先执行 `git rm --cached <文件名>` 取消跟踪，否则仍会被管理

## 远程 git 仓库

目前对于 git 所有操作都是在本地进行的。在开发中显然不能这样的，这时我们就需要一个远程的 git 仓库。远程的 git 仓库和本地的本质没有什么区别，不同点在于远程的仓库可以被多人同时访问使用，方便我们协同开发。在实际工作中，git 的服务器通常由公司搭建内部使用或是购买一些公共的私有 git 服务器。我们学习阶段，直接使用一些开放的公共 git 仓库。目前我们常用的库有两个：GitHub 和 Gitee（码云）

将本地库上传 github：https://github.com/

```bash
# git remote add <remote name> <url>
git remote add origin https://github.com/github-name/git-repository-name.git
# 修改分支的名字的为main
git branch -M main
# git push将代码上传服务器上
git push -u origin main
```

将本地库上传 gitee：https://gitee.com/

```bash
# git remote add <remote name> <url>
git remote add origin https://gitee.com/gitee-name/git-repository-name.git
# git push将代码上传服务器上
git push -u origin master
```

> GitHub 新仓库默认分支是 main，所以推送时将分支改名为 main，Gitee 新仓库默认分支是 master，所以推送时分支名为 master，这是平台默认分支名差异。

基本操作

> 注意：推送代码之前，一定要先从远程库中拉取最新的代码，先 pull 后 push

- `git remote` 列出当前的关联的远程库，查看远程库信息
- `git remote -v` 列出当前的关联的远程库+远程库的详细信息
- `git remote add [自定义远程库名] [url]` 关联远程仓库
- `git remote rename [旧名] [新名]` 修改远程库的名字
- `git remote remove [自定义远程库名]` 删除远程库
- `git push -u [自定义远程库名] [分支名]` 向远程库推送代码，并和当前分支关联，-u是--set-upstream的简写，作用是关联本地分支和远程分支，后续推送/拉取可直接用 git push/pull 无需指定分支
- `git push` 向远程库推送代码，默认推送到和当前分支关联的远程分支，如果本地的版本低于远程库，push默认是推不上去
- `git push [自定义远程库名] [指定本地分支]:[指定远程分支]` 向远程库推送代码，将指定本地分支推送到指定远程分支
- `git clone [url]` 从远程库下载代码
- `git fetch` 从服务器上下载所有代码，但是它不会将代码和当前分支自动合并。要想推送成功，必须先确保本地库和远程库的版本一致。使用fetch拉取代码后，必须要手动对代码进行合并 `git merge [远程库名/远程库分支名]`
- `git pull` 从服务器上拉取代码并自动合并 `git pull = git fetch + git merge`

SSH 公钥和私钥

- 首次和远程进行对接时，会进行账号密码的验证，利用SSH公钥和私钥可以进行免密码登录
- 生成的SSH公钥和私钥内容存放在 `C:\Users\[用户名]\.ssh` 目录下
- `ssh-keygen` 生成 SSH 公钥和私钥，一直回车即可生成
- `ssh-keygen -t rsa` 使用 rsa 算法进行加密生成 SSH 公钥和私钥，一直回车即可生成，官方推荐
- `id_rsa.pub` 公钥，添加至远程库账户中即可
- `id_rsa` 私钥

## github 的静态页面 Github_Pages

在github中，可以将自己的静态页面直接部署到github中，它会给我们提供一个地址使得我们的页面变成一个真正的网站，可以供用户访问。要求：

- 静态页面的分支叫做 `gh-pages`
- 静态页面的文件必须放在主分支的 `docs` 目录中
- 在github中，点击仓库的 `Settings`，找到 `Pages`，选择 `gh-pages` 分支，点击 `Save`，即可获得一个网址，直接登录即可查看：https://github-name.github.io/git-repository-name/

```bash
# 将该分支命名为gh-pages
git branch -M gh-pages

# 推送至origin远程库的gh-pages分支
git push -u origin gh-pages
```

- 如果希望页面可以通过 `xxx.github.io` 访问，则需要将库的名字配置为 `xxx.github.io`
